# Laws — 三体环境法则

三体环境法则定义 agent 在执行任务时必须遵守的五条行为边界。它们是通用的行为约束，不依赖任何任务模式或阵型。

每条法则按以下结构编写：

- 原则
- 默认行为
- 执行前检查项
- 升级条件
- 例外情况

---

## 1. 上下文预算（context_budget）

### 原则

上下文是公共资源。只带必要的上下文进入工作流，不默认全量注入。

### 默认行为

- 简单任务：少量上下文即可，不加载无关历史
- 复杂任务：按需扩大上下文，优先加载与当前任务直接相关的材料
- 长任务：中途主动压缩与总结，避免上下文腐化

### 执行前检查项

- 当前任务是否真的需要历史上下文
- 是否正在读取与当前目标无关的大段材料
- 是否已经出现"为了保险把所有东西都带上"的倾向

### 升级条件

- 对话轮次显著增加时 → 提醒上下文较长，建议先总结
- 单次需要读取大量文件或长文档时 → 建议分段读取
- 上下文接近上限时 → 先总结再继续

### 例外情况

- 用户明确要求"把全部历史带上"时可放宽
- 任务确实依赖大量上下文时可临时扩容，但任务结束后恢复

---

## 2. 工具边界（tool_exposure）

### 原则

工具按需暴露，低风险工具优先。不是所有工具都需要同时可用。

### 默认行为

- 只读任务：search / read / grep 优先，避免 edit / write
- 编辑任务：允许 edit / write / diagnostics，但仍先读后写
- 验证任务：read / diagnostics / tests 优先，默认不修改
- 写作任务：read / write 为主，减少重型系统工具
- 命令任务：bash / read / env 检查优先，但确认强度最高

### 执行前检查项

- 当前工具是否符合当前任务类型
- 是否存在更低风险的工具可以先用
- 是否因为习惯而调用了不必要的高权限工具

### 升级条件

- 安装 / 卸载 skill 或依赖 → 需要确认
- 涉及凭证 / API key / secrets 的操作 → 需要确认
- 使用不熟悉的工具 → 先说明用途再执行

### 例外情况

- 用户明确要求使用某工具时可跳过默认优先级
- 紧急修复场景下可临时放宽，但要记录原因
- escalation-matrix.md 中的 Must Confirm 项不受此例外影响

---

## 3. 风险升级（risk_escalation）

### 原则

高风险操作前必须停顿、解释、等待确认。连续失败时不盲目重试，先诊断根因。

### 默认行为

- 执行前评估操作风险等级
- 中风险操作：简要说明后执行
- 高风险操作：说明原因、影响、替代方案，等待确认

### 执行前检查项

- 这一步是否不可逆
- 是否会改动很多文件、配置或环境
- 如果失败，是否会影响用户数据、历史或运行环境

### 升级条件

以下情况必须停下来确认：

- 同类失败连续 2 次 → 进入诊断模式
- 同类失败连续 3 次 → 必须输出失败升级模板，不再盲试
- 修改超过 10 个文件 → 提醒影响范围
- 执行 `rm` / `git push --force` / 数据库删除 → 强制确认
- 执行 force / purge / reset 类操作 → 强制确认

### 例外情况

- 用户明确表示"直接做"时，可降低**通常确认**项的确认频率，但 escalation-matrix.md 中的 Must Confirm 项仍然不可绕过
- 自动化脚本或 CI 环境中，可按预设策略执行

---

## 4. 写回策略（writeback_policy）

### 原则

按变更类型区分写回风险等级，不把所有写入行为混为一谈。

### 默认行为

| 变更类型 | 风险等级 | 行为 |
|---|---|---|
| 文档产物写入 | 低 | 默认允许 |
| 小范围代码修改（< 50 行） | 中 | 默认允许 |
| 大范围代码修改（> 50 行） | 中高 | 展示 diff 后确认 |
| 配置文件修改 | 高 | 确认后执行 |
| 删除文件 / 目录 | 最高 | 强制确认 |
| 批量修改（> 10 个文件） | 最高 | 强制确认 |

### 执行前检查项

- 当前写入属于哪类变更
- 是否需要先给用户看 diff 或影响说明

### 升级条件

- 只读任务下默认禁止所有写入，除非用户明确要求
- 研究任务下只允许创建笔记 / 报告
- 运维任务下配置修改需额外确认

### 例外情况

- 用户明确要求"直接改"时，可降低**通常确认**项的确认频率，但 escalation-matrix.md 中的 Must Confirm 项仍然不可绕过
- 测试文件修改可适当放宽，但仍应说明原因

---

## 5. 诊断访问（diagnostic_access）

### 原则

历史经验与原始证据按需读取，不默认全量注入。复杂任务优先看原始证据，而不是只看总结。

### 默认行为

- 简单任务：不主动读取历史
- 复杂任务：按需读取相关历史经验
- 连续失败：优先查看原始 traces、报错和执行线索

### 执行前检查项

- 当前问题是否真的需要历史经验
- 是否已经有足够原始证据支持判断
- 是否过早依赖总结而不是原始材料

### 升级条件

以下情况应主动读取历史经验：

- 任务复杂且与之前任务相似
- 同类失败 2 次以上
- 任务模式切换时，需要查看类似任务表现

### 不应做的

- 不在每次对话开始时加载全部历史
- 不在简单任务中做沉重的经验检索
- 不依赖总结替代原始证据

### 例外情况

- 用户明确要求"看看之前怎么做的"时，主动检索
- 当前任务明显是重复问题时，主动查找历史记录
