# Escalation Matrix - 风险升级矩阵

本文档列出 agent 在执行任务时必须升级确认的高风险动作，以及连续失败时必须使用的固定模板。

---

## 必须确认（Must Confirm）

以下操作在执行前必须停下来，说明原因、影响和替代方案，等待用户明确批准。

| 操作类型 | 示例 | 风险说明 |
|---|---|---|
| 删除文件 | `rm -rf`, `git clean -fd` | 不可逆，可能丢失重要数据 |
| 删除目录 | `rm -rf <dir>` | 可能误删整个模块 |
| 强制推送 | `git push --force` | 可能覆盖他人提交 |
| 强制重置 | `git reset --hard`, `git checkout -- .` | 丢失未提交更改 |
| 数据库删除 | `DROP TABLE`, `DELETE FROM` | 不可逆数据丢失 |
| 配置修改 | 修改 `.env`, `config.yaml`, `package.json` | 可能破坏运行环境 |
| 安装依赖 | `npm install`, `pip install` | 可能引入不兼容版本 |
| 安装 skill | 安装新的 agent skill | 可能改变 agent 行为 |
| 部署操作 | `deploy`, `push to prod` | 影响线上环境 |
| 接触凭证 | 读取 `.env`, `credentials.json`, API keys | 可能泄露敏感信息 |
| 批量修改 | 同时修改 > 10 个文件 | 影响范围大，需要确认意图 |
| 系统级命令 | `sudo`, `chmod`, `chown` | 权限变更，可能影响系统安全 |

---

## 通常确认（Usually Confirm）

以下操作通常建议确认，但在上下文非常明确时可直接执行。

| 操作类型 | 示例 | 风险说明 |
|---|---|---|
| 大范围重构 | 重构超过 3 个文件 | 可能引入回归 |
| 跨模块变更 | 同时修改前后端 | 影响面大 |
| 长命令执行 | 执行超过 30 秒的命令 | 可能阻塞或产生意外结果 |
| 修改测试文件 | 修改 `test_*.py`, `*.test.ts` | 可能掩盖真实问题 |
| 修改 lock 文件 | `package-lock.json`, `poetry.lock` | 可能破坏依赖一致性 |

---

## 通常安全（Usually Safe）

以下操作通常可直接执行，无需额外确认。

| 操作类型 | 示例 | 说明 |
|---|---|---|
| 只读操作 | `read`, `grep`, `ls`, `cat` | 不改变任何状态 |
| 小范围代码修复 | 修改 < 50 行代码 | 影响面小 |
| 创建新文件 | 在目标目录下新建文件 | 不覆盖已有内容 |
| 文档写入 | 创建笔记、报告、总结 | 低风险 |
| 运行测试 | `pytest`, `npm test` | 只读验证 |
| 查看日志 | `tail`, `grep` 日志文件 | 只读诊断 |
| 查看状态 | `git status`, `docker ps` | 只读检查 |

---

## Must Confirm 不可绕过

以下操作无论用户是否说"直接做""不用确认""force"，都必须停下来说明原因、影响和替代方案，等待用户**显式批准**（是/确认/继续）：

- 删除文件或目录
- 强制推送或强制重置（`git push --force`, `git reset --hard`）
- 数据库删除（`DROP TABLE`, `DELETE FROM`）
- 批量修改超过 10 个文件
- 接触凭证或 secrets（`.env`, `credentials.json`, API keys）
- 系统级命令（`sudo`, `chmod`, `chown`）

这些是硬确认项，用户催促不改变确认要求。

## 升级后行为

当触发必须确认的操作时，agent 必须：

1. 说明原因：为什么要做这个操作
2. 说明影响：可能带来什么后果
3. 给出替代方案：如果有更安全的做法，先提出
4. 等待批准：在用户明确确认前不继续执行

---

## 连续失败升级

当同类失败连续发生时：

- 连续 2 次：进入诊断模式，停止机械重试
- 连续 3 次：必须输出失败升级模板，等待用户确认下一步

不允许在没有新增诊断信息的情况下重复同样动作。

---

## Failure Escalation Template

当进入失败升级时，统一使用以下格式：

```text
当前步骤：
失败现象：
已尝试动作：
初步判断的根因：
可选下一步：
- 方案 A：
- 方案 B：
- 方案 C：
推荐方案：
是否需要用户确认：
```

要求：

- `当前步骤` 说明失败发生在哪个阶段
- `失败现象` 只写客观现象，不写模糊判断
- `已尝试动作` 列出已经做过的事，避免重复尝试
- `可选下一步` 至少给出 2 个可选方向
- `推荐方案` 必须明确，不允许只丢选项不做判断

## 高风险确认示例

```text
我准备执行以下操作：
  rm -rf /tmp/build-cache

原因：清理旧的构建缓存，确保重新构建使用最新依赖。
影响：如果缓存中有未提交的临时文件，将被永久删除。
替代方案：可以先移动到备份目录，而不是直接删除。

是否继续？
```
