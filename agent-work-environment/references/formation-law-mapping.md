# Formation-Law Mapping - 阵型与法则偏置映射

本文件用于快速回答一个问题：

> 当前已经选定阵型后，五条环境法则应该如何偏置执行？

它不是理念说明，而是速查表。优先用于 session 中的快速判定和下游 handoff 前的边界确认。

---

## 使用方法

当已经判定当前任务属于某个阵型时：

1. 在总矩阵中确认该阵型的法则偏置
2. 根据该偏置决定本轮的读取、写入、确认和诊断强度
3. 如果任务性质变化，切阵后重新查表，而不是沿用旧偏置

---

## 总矩阵

| 法则 ↓ / 阵型 → | 观机阵（研究模式） | 破局阵（实现模式） | 明鉴阵（验证模式） | 立言阵（写作模式） | 行令阵（运维模式） |
|---|---|---|---|---|---|
| 上下文预算 | 宽松 | 中等 | 聚焦 | 中等 | 中等 |
| 工具边界 | 只读优先 | 编辑优先 | 验证优先 | 写作优先 | 命令优先 |
| 风险升级 | 中 | 中高 | 中 | 低 | 最高 |
| 写回策略 | 很保守 | 正常允许 | 默认只读 | 文档允许 | 高确认 |
| 诊断访问 | 按需 | 连续失败时增强 | 高 | 低 | 中 |

---

## 快速决策规则

### 观机阵（研究模式）

- 如果不确定要不要改，默认先不改
- 如果需要更多材料支持判断，优先扩上下文，不优先扩写入权限
- 如果准备写文件，默认只允许笔记、结论、报告

### 破局阵（实现模式）

- 如果已经确认需要动手，优先定位后修改
- 如果改动范围开始扩大，优先提升确认强度
- 如果连续失败，先增强诊断访问，不要机械重试

### 明鉴阵（验证模式）

- 如果目标是判断对错，优先压缩上下文到关键证据
- 如果发现问题，先给发现和风险等级，不默认顺手修
- 如果没有足够证据，不要下结论

### 立言阵（写作模式）

- 如果目标是交付文档，优先结构和可读性
- 如果需要补事实，先少量读取，不扩大系统操作
- 如果开始碰代码或配置，说明已偏离写作模式，考虑切阵

### 行令阵（运维模式）

- 如果要执行命令，先判断是否不可逆
- 如果涉及环境、配置、部署，确认强度默认最高
- 如果只是查看状态，按只读操作处理；如果要变更环境，再升级确认

## 阵型速查

### 观机阵（研究模式）

| 法则 | 执行偏置 | 本轮意味着什么 |
|---|---|---|
| 上下文预算 | 宽松 | 允许读更多相关材料 |
| 工具边界 | 只读优先 | 优先 `search/read/grep` |
| 风险升级 | 中 | 主要防误判，不防大写入 |
| 写回策略 | 很保守 | 只允许结论/笔记类输出 |
| 诊断访问 | 按需 | 只有相似问题时才查历史 |

### 破局阵（实现模式）

| 法则 | 执行偏置 | 本轮意味着什么 |
|---|---|---|
| 上下文预算 | 中等 | 只读必要上下文，避免全量扫仓库 |
| 工具边界 | 编辑优先 | 允许 `edit/write/diagnostics` |
| 风险升级 | 中高 | 扩大改动前要提高确认 |
| 写回策略 | 正常允许 | 小改可直接做，大改先说明 |
| 诊断访问 | 连续失败时增强 | 失败后优先查原始错误和历史经验 |

### 明鉴阵（验证模式）

| 法则 | 执行偏置 | 本轮意味着什么 |
|---|---|---|
| 上下文预算 | 聚焦 | 只看关键证据，不做宽泛阅读 |
| 工具边界 | 验证优先 | 优先 `read/tests/diagnostics` |
| 风险升级 | 中 | 重点防误判 |
| 写回策略 | 默认只读 | 除非用户要求，否则不写入 |
| 诊断访问 | 高 | 优先看原始 traces、报错和历史证据 |

### 立言阵（写作模式）

| 法则 | 执行偏置 | 本轮意味着什么 |
|---|---|---|
| 上下文预算 | 中等 | 够写即可，不做重型检索 |
| 工具边界 | 写作优先 | 以 `read/write` 为主 |
| 风险升级 | 低 | 文档写入默认低风险 |
| 写回策略 | 文档允许 | 可以写文档，不碰代码和配置 |
| 诊断访问 | 低 | 只有延续历史文档时才看历史 |

### 行令阵（运维模式）

| 法则 | 执行偏置 | 本轮意味着什么 |
|---|---|---|
| 上下文预算 | 中等 | 重点补环境和状态信息 |
| 工具边界 | 命令优先 | 允许命令、日志、环境检查 |
| 风险升级 | 最高 | 变更环境前先确认 |
| 写回策略 | 高确认 | 配置和环境改动不默认放行 |
| 诊断访问 | 中 | 出问题时查日志和历史操作记录 |

## 常见路由示例

示例 1：

- 用户输入：“先帮我分析这个线上报错原因，不要改代码。”
- 阵型：观机阵（研究模式）
- 法则重点：宽松上下文、只读优先、很保守写回

示例 2：

- 用户输入：“这个接口确认是空指针了，直接修。”
- 阵型：破局阵（实现模式）
- 法则重点：编辑优先、中高风险、失败时增强诊断

示例 3：

- 用户输入：“review 一下这次改动有没有引入鉴权问题。”
- 阵型：明鉴阵（验证模式）
- 法则重点：聚焦证据、默认只读、高诊断访问

示例 4：

- 用户输入：“把刚才的排查结果整理成团队 SOP。”
- 阵型：立言阵（写作模式）
- 法则重点：文档允许、写作优先、低风险升级

示例 5：

- 用户输入：“把修复后的服务发布到测试环境。”
- 阵型：行令阵（运维模式）
- 法则重点：命令优先、最高风险升级、高确认写回
