# Integrations

本文件定义 `agent-work-environment` 作为总控 skill 时，如何与下游执行 skill 协作。

原则：

- `agent-work-environment` 负责选模式、定边界、决定是否确认或切阵
- 下游 skill 负责真正执行
- 如果任务已明显进入执行阶段，应尽快 handoff，不要让总控层吞掉执行职责

---

## 标准交接格式

当本 skill 决定把任务交给下游 skill 时，建议输出以下结构：

```text
当前阵型：<阵型>
交接目标：<接下来要完成什么>
适用边界：<本轮最重要的 2-3 条边界>
确认要求：<是否必须确认>
完成标准：<下游 skill 至少要交付什么>
推荐 skill：<推荐下游 skill>
```

---

## 与 Code 协作

适用场景：

- 当前阵型是破局阵（实现模式）
- 用户要求“实现”“修复”“改代码”“补功能”

总控层应该做：

- 明确这轮是实现任务，不是继续研究
- 判断是否存在大范围改动风险
- 给出本轮最低完成标准：改动说明 + 验证结果 + 残余风险

交接示例：

```text
当前阵型：破局阵（实现模式）
交接目标：修复接口空指针问题
适用边界：编辑优先；大改动前提高确认；连续失败先诊断
确认要求：若修改超过 10 个文件则先确认
完成标准：改动说明、验证结果、残余风险
推荐 skill：Code
```

## 与 review 协作

适用场景：

- 当前阵型是明鉴阵（验证模式）
- 用户要求“review”“检查”“审查”“找风险”

总控层应该做：

- 明确这轮目标是判断问题，而不是顺手修
- 压缩上下文到关键证据
- 要求下游按发现、证据、风险等级输出

交接示例：

```text
当前阵型：明鉴阵（验证模式）
交接目标：检查本次改动是否引入鉴权风险
适用边界：验证优先；默认只读；高诊断访问
确认要求：无
完成标准：发现、证据、风险等级、修复建议
推荐 skill：review
```

---

## 与 careful 协作

适用场景：

- 当前阵型是行令阵（运维模式）
- 即将执行高风险命令
- 用户明确希望“谨慎一点”“安全模式”

总控层应该做：

- 先把任务切到行令阵（运维模式）
- 明确这是高确认场景
- 把“是否不可逆”作为第一判断项

交接示例：

```text
当前阵型：行令阵（运维模式）
交接目标：清理部署环境中的旧缓存
适用边界：命令优先；最高风险升级；高确认写回
确认要求：涉及删除命令，必须确认
完成标准：操作说明、执行记录、结果、回滚提示
推荐 skill：careful
```

---

## 与 guard 协作

适用场景：

- 当前阵型是行令阵（运维模式）
- 用户不只要谨慎，还要限制写入范围
- 存在生产环境、共享环境或大范围改动风险

总控层应该做：

- 明确这是“高风险 + 高边界约束”场景
- 不只提醒确认，还要建议限制写入范围
- 把“哪些目录可以改、哪些不能改”当成交接内容的一部分

交接示例：

```text
当前阵型：行令阵（运维模式）
交接目标：只在指定目录内处理部署配置
适用边界：命令优先；高确认；限制写入范围
确认要求：修改配置前必须确认
完成标准：操作记录、结果、限制范围说明、回滚提示
推荐 skill：guard
```

## 常见组合路径

### 路径 1：研究 -> 实现 -> 验证

- 观机阵（研究模式） -> `Code` -> `review`
- 适合：先分析原因，再修复，再检查风险

### 路径 2：实现 -> 运维 -> 验证

- 破局阵（实现模式） -> `careful` / `guard` -> `review`
- 适合：修复后要部署，再验证部署结果

### 路径 3：研究 -> 写作

- 观机阵（研究模式） -> `doc` 类 skill
- 适合：调研、分析、方案整理

### 路径 4：验证 -> 实现 -> 写作

- 明鉴阵（验证模式） -> `Code` -> `doc` 类 skill
- 适合：先审查问题，再修复，再写总结

---

## 不该做的事

- 不要让总控层替下游 skill 产出完整执行结果
- 不要在已经明确需要 `Code` 或 `review` 时还停留在抽象路由讨论
- 不要把 `careful` 和 `guard` 当作只是“提醒”，它们是执行层的保护协作者
